Factor

Factors are one of the basic elements of quality models. A factor constitutes a property of the software product (or a part of it) that is related to the product’s quality. A factor is always defined in a way, that it is possible to talk about the degree to which it is present in a product that is assessed. The factor can be further defined by characterizing an entity, but it does not have to define an entity.
The terms attribute and property are used interchangeably in the literature. Here, we chose the term factor to clearly distinguish it from the term quality attribute defined in ISO 9126 [ISO9126].

General Rules

  1. Use a factor to define a property of an entity entity in a textual descriptive way.
  2. Provide textual criteria in the factor's description that defines when an entity exhibits the property and when not.
  3. Create a factor before you think about the measures that quantify it. This is helpful to ensure that you measure what you really want to measure. Furthermore, it helps to create factors at the right level of abstraction, preventing the quantitative explosion of factors and impacts.
  4. There are different types of factors:
    • Product Factor
    • Product Quality Attribute (ISO 25010)
    • Quality in Use Attribute (ISO 25010)
    • Stakeholder Satisfaction
    • Technical Issue

Attributes

Qualified Name

  1. The qualified name identifies the factor; it is generated automatically based on the name and the characterizing entity (if set).

Name

  1. The name of a factor should be an adjective.
  2. If it is not possible to find an adjective, formulate the factor name in a way that the factor can be pronounced as: <entity> must be / must have <name>
  3. Try to use the same name in combination with different entities if you intent to characterize them with respect to common property. A large collection of viable names can be found in the base model.
  4. If there is no specific reason to do otherwise, only captialize the first word in the name.

Characterizes

  1. Specify the entity that is characterized by the factor.

Title

  1. A factor can have an optional title.
  2. The title can be used to provide a more verbose description of the factors, beside the name.

Description

  1. Provide a comprehensive textual criteria in the factor's description that defines when an entity exhibits the property and when not.

Refined by / Refines

  1. Factors are organized in a hierarchy through a refinement relation.
  2. Refinement can only be used between of factors of the same type, e.g., product-factors may only refine product factors.

Influenced by / Influences

  1. Factors may influence each other, which is modeled as an Impact.

Measured By

  1. Factors can be quantified by one or more measures.

Product Factor

Product factors are properties of specific parts of the software product. These parts are specified as artefacts or product entities, such as, Conciseness @Identifier or Appropriateness @Comment.
  1. Use product factors to characterize parts of the software.
  2. A product factor must make sense, without referring to the impact.
    Positive example: [Structuredness @Expression].
    The structuredness is defined, independent of the impact it has on understandability.
    Negative example: [Efficiency @Expression].
    You should not introduce this factor and use it to collect all measures that have an impact on performance. Not doing so, assures that a new measure can be uniquely assigned to a factor, without having in mind its impact.
  3. An impact of a product-factor to a product quality attribute or a quality in use factor should point to a leaf in the hierarchy.
    This way it is assured that a justification is given for each of the impacted factors.
    Example: Instead of modeling an impact to [Maintainability] you should model two impacts to [Analyzability], [Modifyability] and give a special justification for both of them. If you would only model one impact to [Maintainability] the justification is less clearer.

Product Quality Attribute (ISO 25010)

Product quality attributes define one way to decompose the abstract concept software quality. These quality attributes, as given in ISO 25010, relate to the quality of the product without explicitly considering its use. These quality attributes are colloquially called -illities, because they contain, for example, reliability or maintainability. In the standard, the top level attributes are refined by so-called quality characteristics.
  1. Product quality attributes model the -ilities of the ISO 25010.
  2. These factors do characterize the entity Product.

Quality in Use Attribute

Quality in use attributes define a way to decompose the abstract concept software quality. These quality attributes are defined in the ISO 25010 as description of the quality in its various forms of usage. Examples are efficiency or effectiveness. To be more precise, we add the activity that is characterised by the attribute as entity to the factor. Activities like maintenance, program comprehension, modification, or testing, which can be decomposed in their respective sub-activities, provide a means to model software development cost structures.
  1. The entity of a quality in use attribute is an activity.
  2. Characterize an activity by:
    • Effectiveness
    • Efficiency
    • Safety
    • Satisfaction

Stakeholder Satisfaction

On a high level of abstraction, product quality contributes to the satisfaction of specific stakeholders. This can also be modeled using the stakeholder as an entity.
  1. The entity of a stakeholder satisfaction is a stakeholder.
  2. These factors have always the name "Satisfaction".

Technical Issue

Technical issues are also a way to decompose the abstract concept of quality. This is a technical view, which assigns problems to areas such as memory or declaration.
  1. TIC factors do not characterize specific entities.
